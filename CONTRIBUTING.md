\# Contributing



Welcome to Ship Relight Guardian.



This repository explores modular, safety-oriented decision architectures for engine relight following an engine-out event.



\---



\# Mission



Build software that is:



\- Modular

\- Explainable

\- Testable

\- Safety-first

\- Simulation-driven

\- Easy to understand



\---



\# Design Principles



Every module should have a single responsibility.



Every decision should be explainable.



Every safety gate should be documented.



Every output should be traceable to its inputs.



No hidden logic.



No "magic numbers."



No undocumented assumptions.



\---



\# Coding Standards



Keep functions small.



Prefer readable code over clever code.



Document every public function.



Avoid unnecessary dependencies.



Write modular components.



\---



\# Pull Requests



Every pull request should answer:



1\. What problem does this solve?

2\. Why is this solution correct?

3\. How was it tested?

4\. Does it affect another module?



\---



\# Documentation



Every new module must include:



\- Purpose

\- Inputs

\- Outputs

\- Decision logic

\- Failure modes



\---



\# Repository Philosophy



The architecture comes first.



Implementation comes second.



Safety comes before performance.



Understanding comes before optimization.



\---



\# Public Boundary



This repository is a public educational and simulation project.



It does not contain:



\- proprietary flight software

\- deployable launch vehicle control code

\- confidential telemetry

\- vendor-specific implementation details



The goal is to demonstrate engineering methodology, modular software design, and safety-oriented decision making.
